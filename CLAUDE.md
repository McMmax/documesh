## Architecture

Nuxt 4 (srcDir = `app/` by default) — all source lives under `app/`, not the root:

```

│   ├── useYjsProvider.ts        # Y.Doc + IndexedDB persistence; SpacetimeDB transport wired in Phase 3

├── layouts/

├── pages/
```

### Data Flow

```
SpacetimeDB (truth) ←→ Y.js (CRDT sync) ←→ TipTap (editor)
                              ↓
                     IndexedDB (local persistence)
```

- **SpacetimeDB** holds canonical document state (`Document` table, JSON content via `YjsUpdate` rows, presence via `AwarenessState`).
- **Y.js** handles CRDT merges between collaborators. The `useYjsProvider` composable owns the `Y.Doc` — one per open document, never recreated.
- **TipTap** binds to the `Y.Doc` via `Collaboration` extension. The `useEditor` composable returns the extensions array; the editor instance lives in `EditorCore.vue`.
- **IndexedDB** (`y-indexeddb`) provides offline persistence until the SpacetimeDB transport connects.

### SpacetimeDB Access Pattern (v2)

SpacetimeDB 2.x uses a **provider pattern** — not a Nuxt plugin. The `spacetimedb` package (v2, replacing `@clockworklabs/spacetimedb-sdk`) ships first-class Vue composables at `spacetimedb/vue`.

**Phase 3 setup** (after `spacetime generate --lang typescript --out-dir app/module_bindings`):
```ts
// app/app.vue — wraps the whole app
import { DbConnection } from '~/module_bindings'
import { SpacetimeDBProvider } from 'spacetimedb/vue'
const builder = DbConnection.builder()
  .withUri(cfg.spacetimedbUri)
  .withDatabaseName(cfg.spacetimedbModule)
```

**In components:**
- `useSpacetimeDB()` — returns `{ isActive, identity, token, getConnection }` from `spacetimedb/vue`
- `useTable(query, callbacks?)` — reactive `[rows, isReady]` tuple, scoped query required
- `useReducer(reducerDef)` — returns async function to call a reducer

Never subscribe to a full table — always filter by user or document ID. `useTable` and `useReducer` require `<SpacetimeDBProvider>` above them in the component tree.

### TipTap Init Pattern

Editor is always initialized inside a composable (`useEditor`), never inline in a component. `EditorCore.vue` calls `useEditor`, wraps the mount in `<ClientOnly>`, and calls `editor.destroy()` in `onBeforeUnmount`. Custom extensions live in `app/components/editor/extensions/` (one file each).

### Inspira UI

Not an npm package — components are copy-pasted into `components/ui/` and modified structurally only when props/slots can't cover the need. Never modify a copied component in place without renaming it.

## Hard Rules — Never Break These

- `<script setup lang="ts">` always. Options API is banned.
- `shallowRef` for editor state and large objects. `ref` for primitives only.
- `useFetch` / `useLazyFetch` for data fetching. Never raw `fetch` from a component.
- `<ClientOnly>` wrapper with `#fallback` slot on TipTap editor and all SpacetimeDB subscriptions.
- Tailwind class order follows Prettier plugin. No `@apply` in component `<style>` blocks.
- Store document content as **JSON**, not HTML, in SpacetimeDB.
- SpacetimeDB connection via `<SpacetimeDBProvider>` in `app/app.vue` — never in a plugin. Use `useTable`/`useReducer` from `spacetimedb/vue`.
- Subscriptions scoped by user or document — never full table subscribes.
- Destroy TipTap editor in `onBeforeUnmount`. No exceptions.
- No manual imports of Vue primitives or Nuxt composables — auto-import handles them.
- Autosave debounce floor is 1000ms. Never save on every keystroke.
- Never put ProseMirror state, TipTap JSON, or `Y.Doc` contents in `ref` or `reactive`.

## Tone

Terse. No docstrings on internal functions. No filler comments. Code reads like a senior engineer wrote it, not a tutorial.

### Generated-Looking Patterns — Strictly Banned

The web-style-guide defines the standard. These patterns specifically signal AI output and are prohibited:

- Verbose variable names: `currentActiveDocument`, `handleButtonClickEvent` — use `doc`, `onClick`
- Narrating comments: `// Loop through each item and add to array` — say nothing or say why
- Boilerplate guards for impossible cases: `if (!arr) return []` on an array you just constructed
- Helper wrappers that add no logic: `function getTitle(doc) { return doc.title }`
- JSDoc / block comment headers on internal functions
- Over-structured error handling: `try/catch` with `console.error` on code that can't throw
- Symmetrical but unnecessary code: `if (x) doX(); else doNotX()`
- Trailing summaries or "Here's what I did" prose in any file
- Anything that reads like it's explaining itself to a beginner
